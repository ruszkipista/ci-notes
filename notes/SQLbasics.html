Because it is an interpreter, the Oracle system responds directly to your queries written at the MySQL prompt, so you immediately see and can check the results of them.  You should always check your results, as it is much more likely that you will make logical errors than syntax errors.

Each time you type in an SQL statement, it is stored in a buffer overwriting your previous statement.  In the lab session next week you will be editing and saving statements.

A few notes about the MySQL environment and its special commands

SQL statements are terminated by a semicolon ; 
A statement can be written on more than one line.  Line numbers are displayed for the second and succeeding lines.  Don't forget the semicolon at the end of the statement ;

SQL Statements are stored in a buffer.  The contents of the buffer can be viewed by typing:

list (or l for short)

The contents of the buffer can be run by typing:

run  (or just / )

Selecting Columns

In SQL, we can select what columns we want to appear. For example

SELECT ename, sal FROM emps;

In English: Select the columns called ename and sal in that order from the emps table.

We can rearrange the order of columns

SELECT sal, ename FROM emps;

In English. Select the columns called sal and ename, in that order from the emps table.

Select DISTINCT

Often, columns in tables will contain elements which repeat. For example, in the emps table, many employees work in the same department. Therefore, if you look down the deptno column, the same values appear repeatedly. If we say

SELECT deptno FROM emps;

In English:  Select the column called deptno from the table emps.

Then the deptno from every row in the emps table will be displayed

If we want to get back each deptno only once, we say

SELECT DISTINCT deptno FROM emps;

In English: Select every unique value in the deptno column from the emps table.

DISTINCT works across all the columns that are being returned. For example, if we say SELECT DISTINCT job, deptno FROM emps, then every unique combination of job and deptno is returned.

SELECT DISTINCT job,deptno FROM emps;

In English: select every unique combination of job and deptno from the emps table.

Ordering

By default, SQL returns rows in the order in which they are stored in the table. If we wish to return rows in a particular order, we use ORDER BY. The ORDER BY clause comes at the end of the SQL statement.

SELECT ename, job, sal FROM emps ORDER BY sal DESC;

In English: Select the ename, job and sal column from the emps table. Sort the rows returned in descending order of salary.

If we want to order in ascending order, than in the previous statement, we would change DESC to ASC, so the statement would read SELECT ename, job, sal FROM emps ORDER BY sal ASC;

We can order by multiple columns. In this case, the rows returned are ordered by the first column mentioned, then by the second column mentioned.

SELECT ename, deptno, sal FROM emps ORDER BY deptno, sal;

In English: Select ename, deptno and sal from the emps table. Order the rows returned by deptno, then order rows by salary.

Formatting Output

Often in SQL, we may wish to alter the output, in order to present reports and data professionally. The kinds of alterations we may like to do can be broken into a number of different types.

Arithmetic Operations

Sometimes, we might like to be able to perform calculations on rows that are returned. We can use + - / * operators. For example, Suppose we wish to calculate what the salary of every individual would be if we doubled employees salaries

SELECT ename, sal * 2 FROM emps;

In English: Select the name and salary from every row in the emps table. Multiple the salary in each row by 2 and display the name and salary multiplied by 2.

Manipulations

SQL offers a variety of functions we can use to manipulate fields when they are returned. These functions are applied in the same ways. Where we normally put in the fieldname, we surround the fieldname with the function.

The name display functions available in this category include

  • lower () – converts all characters to lower case; i.e. JONES → jones
  • upper() – converts all characters to upper case; i.e jones → JONES
  • initcap() – convert the first character to upper case and all other characters to lower case; i.e.  JONES → Jones

SELECT lower(ename) FROM emps;

In English: Select the ename field from the emps table. For every ename returned, convert it to lower case.

The round function is slightly different, in that it takes two parameters, the field to be manipulated and the number of decimal points to appear after the point. For example,

SELECT round(sal, 2) FROM emps;

In English: Select all the salaries from the emps tables. For each salary returned, set it so that there are only two numerals after the decimal point.

Column Manipulations

SQL has a variety of commands that allow us to manipulate the columns that are returned. Firstly, we can set the column heading. For example, suppose we enter the two following commands

COLUMN deptno HEADING 'Department Number'
SELECT deptno FROM emps;

In English: Every time that deptno appears in the heading, replace it with the string Department Number. So, when we run the second SQL command, the heading deptno is replaced with the string 'Department Number' in the result.

Sometimes, we might like to concatenate (join) two (or more) fields together. To do this, we use the || operators.

SELECT ename || job FROM emps;

In English: This selects the ename and job field from the emps tables. For every row the ename and job are combined into one column. This column is returned. In other words, this will return:

JONESANALYST rather than JONES, ANALYST

Sometimes, we might like to add in text into the returned rows. We do this by surrounding the text to be inserted with single quotes and  inserting it into the SQL statement like so,

SELECT ename, ' is a ', job FROM emps;

In English: select the ename and job field columns from the emps tables. In each row, between the ename field and the jobs field, insert the text ' is a '

WHERE

So far, every query we have looked at returns back every row in the table.  Often, we are only interested in subsets of the data. For example, we might only be interested in the salesmen in the emps table. To return back only the rows that contain salesmen, we say

SELECT * FROM emps WHERE job='SALESMAN';

In English. We examine the rows in the table. We select every row where the job column contains the value 'SALESMAN'.  Then, we return back every column from the rows that we have selected.

Job='SALESMAN' is a predicate, which means that for a particular row, it will be true (the row will contain SALESMAN in the job column) or false (the row will contain something other than SALESMAN in the job column).

We can negate predicates by using NOT. For example

SELECT ename, deptno FROM emps WHERE NOT job='SALESMAN';

In English We examine the rows in the table. We select every row where the job column does not contain the value SALESMAN. We return the ename and the deptno from the rows selected.

In this case, we find every row where the job column does not contain SALESMAN

We can combine predicates, using AND and OR. For example, suppose we wish to find salesmen who work in department number 20. In this case, we have two predicates that must be true i) job='SALESMAN', ii) deptno=20. We combine them using AND, like so

SELECT ename FROM emps WHERE job='SALESMAN' AND deptno = 20;

In English We examine the rows in the table. We select every row where the job column contains the value SALESMAN and the deptno column contains the value 20. From the selected rows, we take the ename column

Note, we don't put 20 in quotes, because it is a number. When we are checking numbers, we leave out the quotes.

We can use all the standard relational operators (>, <, <=, >=, =, != etc) in the WHERE clause.  For example to find salesmen who earn more than 1000, we say

SELECT ename FROM emps WHERE job='SALESMAN' AND sal > 1000;

In English: We examine the rows in the table. We select every row where the job column contains the value SALESMAN and the value in the sal column is greater than 1000. From the selected rows, we return the ename.

There are some special SQL keywords that we can use in the WHERE clause. The first is the NULL keyword. Sometimes, columns will have a NULL value. This is not the same as 0. For example, in the emps tables, employees who are not salesmen will have a NULL value in the comm field. We can use the NULL keyword to check does a column contain a NULL value. For example.

SELECT ename, job FROM emps WHERE comm IS NULL;

In English: We examine all the rows in the table. We select every row where the comm column contains the value NULL. From the selected rows, we return the ename and job column.

When we are checking strings, the standard relational operators are too limiting. Sometime, we might like to say things like, find all the employees whose name starts with the letter J. We can use the LIKE keyword to allow us to do this. We could say

SELECT ename, job FROM emps WHERE ename LIKE 'J%';

In English: We examine all the rows in the table. We select every row where the ename column contains a value that start with the letter J, and is followed by 0 or more of any other character. From the selected rows, we return back the ename and the job columns.

Dates are stored as a particular type in SQL. In order to manipulate dates, we have two functions available to us. to_date converts a string to a date, while to_char converts a date to a string. For example

SELECT ename, deptno FROM emps WHERE hiredate > to_date('01-Feb-1981') ;

In English: We examine all the rows in the table. Using the to_date function we convert the string '01-Feb-1981' in a date that can be used by SQL. Then, we select every row where the value in the hiredate column is greater than the value we calculated. From the rows selected, we return back the ename and the deptno columns.

UNION, INTERSECT, MINUS

An SQL query returns back a result. For example, the query SELECT ename, job FROM emps returns back a result that has 2 columns (ename, job), and 14 rows (since there are 14 rows in the database.

Different queries will return back different results. For example, SELECT ename, job, sal FROM emps will return back relation that has 3 columns and 14 rows, and so on.

If we have two separate queries, we can perform operations on the results of those two queries provided the results are comparable. By comparable, we mean that the results have the same number of columns, and the type of the columns correspond.

For example, suppose we have a query

Query1 : SELECT ename, sal FROM emps WHERE job='PRESIDENT';

This would return

KING, 5000

And we have another query

Query2: SELECT ename, sal FROM emps WHERE job='MANAGER';

Which would return

JONES, 2975

BLAKE, 2850

CLARK, 2450

These two results are comparable, since they have the same number of columns (2), and the types of the columns correspond. In this situation, we can combine the result generated by Query1 with the result generated by Query2 by saying

SELECT ename, sal FROM emps WHERE job='PRESIDENT'
UNION SELECT ename, sal FROM emps WHERE job='MANAGER';

Consider another example.

SELECT ename, sal FROM emps WHERE ename='JONES'
UNION SELECT ename, sal FROM emps WHERE job='CLERK';

In English: There are two separate queries in the example above. First, we calculate the results of the first query (highlighted in blue). Then, we calculate the result of the second query (highlighted in red). Then we get the two results and combine them to get the final answer.

The following SQL will not work, because the results generated by the queries will not be comparable.

SELECT ename FROM emps WHERE ename='JONES'
UNION SELECT ename, sal FROM emps WHERE job='CLERK';

The first query (highlighted in blue) will return 1 column, while the second query (highlighted in red) will return 2 columns. Since these results have different numbers of columns, it is not possible to combine them

The other two relevant operators are INTERSECT and MINUS. As with UNION, both of these operators require two comparable results. INTERSECT returns the rows that both results have, while MINUS subtracts the rows in the second results from the first result.

Joins

So far, all the examples we have looked at have used only one table. SQL allows us to use multiple tables. The simplest example of this would be the following query.

SELECT * FROM emps, depts;

Every column in the emps table will be included and every column in the depts table will be included. This means there will be 8 + 3 = 11 columns.  This query will take every row in the emps table and match it with every row in the depts table. For our tables, this means we will get back 14 X 4 = 56 rows.  The final result will have 11 columns and 56 rows. Note that both the emps table and the depts table have a column called deptno. Since we need to be able to uniquely identify each column, the table name of the source table is prepended to the column name, hence, emps.deptno is the deptno column from the emps table, while depts.deptno is the deptno column from the depts. table.

Aggregation Functions

We have looked at five aggregation functions – min, max, count, sum, avg. These functions perform an operation on a column. It is important that you understand how SQL processes these functions. Consider the example

SELECT sum(sal) FROM emps;

In English: First we examine all the rows in the table. Then, we get the sal column. Then we sum the sal column together. Then, we return back the result.

Consider another example.

SELECT sum(sal) FROM emps WHERE job='SALESMAN';

In English: First we examine all the rows in the table. We only select rows where the job column contains the value salesman. From the rows selected, we take the sal column. Then we add all the values together.

It is very important to you understand SELECT ename, max(sal) FROM emps; will not work and why it doesn't works. When you try to execute this query, what happens is as follows. First, we get back all the rows from the emps table. Then, we select the ename column and sal column. Next, we apply the max function to the sal column. This adds all the values together and gives us back 1 result. The problem now arises, how do we convert ename to 1 row? Since we haven't told SQL how to do this, it can't figure it out and so causes an error message:

Another important thing to understand is that when a query is resolved, the WHERE condition is resolved before the aggregation function is implied. If we say SELECT sum(sal) FROM emps WHERE job='MANAGER', the steps that are followed are

  1. SQL gathers all the rows where job='MANAGER' (call this Result1)

        

  1. SQL takes the sal column from Result1 (call this Result2)

  1. SQL calculates sum of the sal in Results2

        

Group By

Aggregation functions work by combining columns together i.e. sum add all the values in a column together, max selects the largest value in a column, etc. Sometimes, what we would like to do is segment a table before we apply an aggregation function. For example, consider the query SELECT sum(sal) FROM emps GROUP BY deptno;

When we execute this function, you can think of it as first breaking the table up into separate subtables, one subtable for each unique value in the field that is being grouped by. Then, the aggregation function is applied to each subtable. Finally, the results are combined again:

             

As before, the WHERE clause is applied before the GROUP BY breaks the table up into separate clauses. So, if we say SELECT max(sal) FROM emps WHERE sal > 1000 GROUP BY deptno; the process that is followed is:

  1. We get back the rows where the salary is greater than 1000

        

  1. We break the table up into subtables, one subtable for each unique value in deptno

        

  1. We get the sal column from each subtable
  2. We calculate the maximum salary in each column

        

  1. We combine the result

        

Another important note is that we can say SELECT deptno, max(sal) FROM emps WHERE sal > 1000 GROUP BY deptno now. When we break the main table up into subtables, in each subtable the value in the deptno column will be same in every row. Thus, SQL can know how to combine the deptno column together. However, if we were to say SELECT ename, deptno, max(sal) FROM emps WHERE sal > 1000 GROUP BY deptno, we would get an error. To see why this is so, look at one of the subtables above again. SQL knows how to combine the sal column, because we have told it. SQL knows how to combine the deptno column, because they are all the same. SQL doesn't know how to combine the ename column however, because there are different values in each row, and we didn't specify how to combine it.

HAVING

We have seen how GROUP BY operates, and we have also seen how the WHERE clause gets applied. So, consider again the second example presented in GROUP BY.

In plain English, this returns for each department the sum of the salaries paid out by that department. Suppose that we are only interested in finding departments whose salary bill was greater than 4000. Our first attempt at writing a query to solve this might be

SELECT sum(sal) FROM emps WHERE sum(sal) > 4000 GROUP BY deptno;

This will not work. It should be obvious now why it won't work. The predicate in the WHERE clause is sum(sal) > 4000. The problem is that if we use WHERE we are trying to apply this predicate, (i.e. use the value sum(sal)) before we have calculated it.

In order to filter on aggregated values, we will have to apply the predicate after the value has been calculated. In order to do this, we use the keyword HAVING. This tells SQL that this predicate is to be applied after the aggregated functions have been calculated. Consider the following statement.

SELECT max(sal) FROM emps WHERE sal > 1000 HAVING max(sal) > 4000 GROUP BY deptno;

In English

  1. Take all the rows
  2. Select all the rows where sal > 1000 (This is applying the WHERE condition). Call this Result1
  3. Break Result1 up into subtables, one subtable for each unique value in deptno

        

  1. Get the sal column for each subtable
  2. Calculate the maximum salary in each subtable

        

  1. Combine the result from each subtable. Call this Result2

        

  1. From Result2, select every row where max(sal) > 4000 (This is applying the HAVING clause)

Simple Subqueries

So far, we have looked at queries in the form SELECT * FROM emps WHERE sal > 1000. If we examine the predicate (sal > 1000) in more detail, then we see that it contains three items. It contains a fieldname (i.e. sal), it contains an operation (i.e. >) and it contains a value (i.e. 1000).

Subqueries allow us to replace the value part of a predicate with a query. Consider the following query : SELECT max(sal) FROM emps; In plain English, this will return us back the maximum salary from the emps table. The final result of this query will be a single value, or if you prefer, a 1 X 1 result.

Since this query will return us back a single value, we can use this query instead of the value in a predicate. Conceptually, you can think of it as working as follows. First, we execute the subquery, then when that has finished and returned a value, we execute the outer query.

Consider

SELECT ename FROM emps WHERE sal = (SELECT max(sal) FROM emps);

In English

  1. We execute the subquery first (i.e. SELECT max(sal) FROM emps;)
  2. This query execute in the normal manner and returns a value, in this case 5000
  3. The outer query then executes, using the value returned. Conceptually, you can think of it as executing
    SELECT ename FROM emps WERE sal = 5000;

Relational operators such as =, >, < etc work by comparing one value against another value. So, when we say sal > 1000, we are saying is the value in the sal column greater than 1000. From the above example, we can see how we can replace a value with a query provided the query returns back only one value. So, we can use SELECT ename FROM emps WHERE empno = 7654, because this returns back only one value (MARTIN). However, consider the query SELECT empno FROM emps WHERE job = 'CLERK'. This will return us back four values (7369, 7868, 7900, 7934). We now have a problem. Saying something like empno = (7369, 7868, 7900, 7934) is nonsensical, since (7369, 7868, 7900, 7934) is not a single value.

SQL does provide us with some additional operators that we can use when we get back multiple results however. The first is the IN operator. The IN operator checks to see if the value on the left hand side is IN the set of values on the right hand side. So,

7900 IN (7369, 7868, 7900, 7934)

will be true, while

7844 IN (7369, 7868, 7900, 7934)

Will be false.

We can use this operator to check values. Consider the following example

SELECT ename FROM emps
 WHERE empno IN (SELECT empno FROM emps WHERE job = 'SALESMAN');

In English:

  1. We run the subquery (in red) first.
  2. This query will return us back 7499, 7521,7654,7844
  3. Next, we run the outer query. We can think of this as being
     SELECT ename FROM emps WHERE empno IN (7499, 7521, 7654, 7844)
  4. This second query will return the ename column from every row where the value in the empno column is in (7499, 7521, 7654, 7844)

Another useful operator is ALL. ALL allows us to say something about all the elements in a set. For example, consider the query SELECT sal FROM emps WHERE job='MANAGER'. In plain English, this returns us the salary of all employees who are managers. The actual values returned will be (2975, 2850, 2450).

A predicate of the form

3000 > ALL (2975, 2850, 2450) 

Will be true, since 3000 is greater than ALL the values. Similarly, a statement

2800 > ALL (2975, 2850, 2450)

Will be false, since 2800 is not greater than 2975, so its not greater than all the values in the set.

So, consider the following example

SELECT ename FROM emps

  WHERE job='MANAGER' AND sal >= ALL(SELECT sal FROM emps WHERE job='MANAGER');

In English

  1. Run the subquery (in red) first.
  2. This query will return us back 2975, 2850, 2450
  3. Next, run the outer query. We can think of this as being SELECT ename FROM emps WHERE job = 'MANAGER' AND sal >= ALL (2975, 2850, 2450)
  4. This query takes all the rows
  5. The first predicate is applied, and it selects only the rows where the value in the job field is equal to MANAGER. Call this Result 1
  6. The second predicate is applied to Result1. This returns back all the rows where the value in the sal column is greater than or equal to all of the values in the set (2975, 2850, 2450).Call this Result2
  7. We return the column enames from Result2

Correlated Subqueries

Simple subqueries operate sequentially, which is to say that the subquery (or subqueries) are performed first, then the main query is executed. Correlated subqueries are intertwined. A good analogy from programming is that correlated subqueries work like a loop inside of loop. Consider the following example of a simple subquery.

SELECT ename FROM emps WHERE sal = (SELECT max(sal) FROM emps);

In English The subquery (in red) returns back the maximum salary from all employees. The outer query then returns the name of the employee whose salary is equal to the result of the subquery, i.e. the maximum salary of all employees. The whole query returns back the ename of the employee with the maximum salary.

Suppose we want to modify the above query so that it returns back the ename of the employee who earns the most in their department. We have seen other ways of doing this, but here we want to explore how to do it with a correlated subquery. A simple algorithmic description of how we would do it would be.

For each employee e

        Find the maximum salary paid out in the department where e works (deptno = deptno)

        If the salary of the e (e.sal) =  maximum salary paid out, then keep e

Move to the next employee

The important point to note is this – we run a separate subquery for every single employee. The SQL to implement this would be

SELECT ename FROM emps E
 WHERE sal = (SELECT max(sal) FROM emps f WHERE e.deptno = f.deptno)

In English

  1. Go to the first row of the emps table (call this instance e).
  2. Get the value of the deptno column in instance e (in this case 20)
  3. Run the subquery for the first time. The subquery is SELECT max(sal) FROM emps f WHERE 20 = f.deptno)
  4. This will return the value 3000
  5. Return to the outer loop
  6. For the first row of the table, sal will not be equal to the 3000, so this row will not be returned.
  7. Move to the second row of the emps table (instance e)
  8. Get the value of the deptno column in instance e (in this case 30)
  9. Run the subquery for the second time. The subquery is SELECT max(sal) FROM emps f WHERE 30 = f.deptno)
  10. This will return the value 2850
  11. Return to the outer loop
  12. For the second row of the table, sal will not be equal to 2850, so this row will not be returned.
  13. Move to the third row of the emps table (instance e)
  14. Repeat sequence until we run out of rows in instance e

The EXISTS keyword is often used with correlated subqueries. Sometimes, queries will return no results. For example, if we execute the query SELECT * FROM emps WHERE ename = 'PADDY', we will get no result back, since there is not employee whose name is PADDY.

If we run a query, and the query return no result, then EXISTS will be false. So, for example,

EXISTS (SELECT * FROM emps WHERE ename = 'PADDY')

will be true. On the other hand

EXISTS (SELECT * FROM emps WHERE ename = 'JONES')

will be false.

We can use EXISTS in the WHERE clause of an SQL statement. For example

SELECT ename FROM emps e
  WHERE EXISTS (SELECT * FROM emps f WHERE e.empno = f.empno AND f.comm IS NOT NULL)

Views

Use views to store the results of intermediate steps in the more complex queries. A view is a virtual relation which represents the result of a query. Once created, a view can be used as a relation in SQL queries.

To create a view, execute the SQL statement: CREATE VIEW <name> AS <query>;
